Sunwave Hub

Plateforme communautaire dédiée à l'univers retrowave & synthwave. Radio live, galerie visuelle générée par IA, soumission de playlists et espace événementiel — le tout déployé en static sur GitHub Pages.

Frontend API YouTube Supabase Vite GitHub Actions Gemini API GitHub Pages
01 — Vue d'ensemble

Pourquoi ce projet ?

Contexte

La communauté retrowave existe principalement sur Discord, Reddit et YouTube — sans espace centralisé qui réunisse musique, visuels et actualités. Sunwave Hub est pensé comme ce point de convergence.

Objectif

Créer une expérience immersive à forte identité visuelle, entièrement statique (zéro serveur à maintenir), avec une base de données légère via Supabase et un pipeline de génération de contenu automatisé.

Contrainte principale

Site statique hébergé sur GitHub Pages — aucun backend custom. Tous les besoins dynamiques (soumissions, playlists) transitent par Supabase via l'API REST côté client.

Résultat

Une SPA sans framework, responsive, avec un design system retrowave cohérent — déployée automatiquement à chaque push sur main via GitHub Actions.

02 — Fonctionnalités

Ce que fait le site

Radio Live — Obsidian Receiver '84

Un lecteur radio custom à l'esthétique CRT/synthwave, piloté par l'YouTube IFrame API. Trois chaînes (Neon Drift, Sunset Vibes, Cyberpunk Nights) correspondant à des playlists YouTube curées. Affichage de la fréquence, du nom de station, du titre en marquee, et VU-mètres animés.

YouTube IFrame API Vanilla JS ES Modules AudioContext API

Galerie Visuelle — Génération Automatisée

Les images de la galerie sont générées automatiquement chaque semaine via un workflow GitHub Actions qui appelle l'API Gemini (Google). Le script génère des visuels rétrowave avec des prompts contrôlés, les commit directement dans le repo, et déclenche un redéploiement. Lightbox intégré avec téléchargement.

Gemini API GitHub Actions (cron) GitHub Pages

Soumission de Playlists

Formulaire de soumission (URL Spotify ou YouTube + titre) persisté dans Supabase via le client JS côté navigateur. Les 4 playlists les plus récentes approuvées sont affichées dynamiquement sur la page d'accueil. Schéma dédié sunwave dans Supabase pour l'isolation.

Supabase (PostgreSQL) @supabase/supabase-js Row Level Security

Design System Retrowave

Design entièrement custom : palette sunset (pink → orange → yellow), overlays scanlines CRT, grille rétro-perspective animée, glassmorphism, typographie Audiowide + Oxanium. Variables CSS centralisées, classes utilitaires sunset-bg / sunset-text / sunset-static pour une cohérence à travers tout le site. Responsive mobile-first.

Tailwind CSS (CDN) CSS Custom Properties Google Fonts Feather Icons
03 — Stack Technique

Technologies utilisées

Build
Vite Bundler

Build rapide, injection des variables d'environnement (VITE_) pour Supabase, sortie statique dans /dist.

Frontend
HTML + CSS + JS Vanilla

Aucun framework UI. ES Modules natifs. Tailwind via CDN pour la rapidité de développement.

Base de données
Supabase PostgreSQL

Schéma isolé sunwave. Client JS direct depuis le navigateur. Anon key + RLS pour sécuriser les écritures.

Audio
YouTube IFrame API Streaming

Lecture de playlists en arrière-plan (iframe 0×0). Gestion des états, changement de station à chaud, récupération du titre en cours.

Contenu IA
Gemini API Génération image

Workflow Actions hebdomadaire. Génère 6 visuels retrowave, les commit dans /public/images/gallery/ et redéploie.

CI/CD
GitHub Actions Deploy

Deux workflows : déploiement sur push main → GitHub Pages. Génération galerie sur cron hebdomadaire.

Pipeline CI/CD
Push sur main

Le workflow deploy.yml se déclenche. Node 20 installé, npm install && npm run build exécuté. Variables Supabase injectées depuis les GitHub Secrets (VITE_SUPABASE_URL, VITE_SUPABASE_ANON_KEY).

Build Vite

Vite bundle le JS (radio.js + imports Supabase), génère les assets optimisés dans /dist. Le HTML et les images sont copiés tels quels.

Déploiement Pages

upload-pages-artifact + deploy-pages publient le contenu de /dist. Aucun serveur, CDN GitHub uniquement.

Cron hebdomadaire — Galerie IA

generate-gallery.yml s'exécute chaque semaine. Appelle l'API Gemini avec des prompts retrowave définis, enregistre les PNGs dans /public/images/gallery/, commit et push → déclenche automatiquement le pipeline de déploiement.

05 — Choix techniques

Pourquoi ces décisions ?

Pourquoi aucun framework (React, Vue…) ?
Le site est principalement du contenu statique avec deux points dynamiques simples (radio + Supabase). Un framework aurait ajouté de la complexité (virtual DOM, state management, hydration) pour un gain nul. Le JS vanilla ES Modules suffit, le bundle final est plus léger, et le déploiement reste un simple dossier de fichiers plats.
Pourquoi Tailwind en CDN plutôt qu'installé ?
Le design utilise abondamment des valeurs arbitraires Tailwind (text-[9px], tracking-[0.5em], bg-pink-500/10…). En CDN, toutes les classes arbitraires sont générées à la volée sans configuration. En production, on basculerait sur Tailwind CLI avec content configuré pour purger les classes inutilisées — mais pour ce projet, la simplicité prime.
Pourquoi Supabase plutôt que Firebase ou une API maison ?
Supabase expose une API REST auto-générée depuis PostgreSQL, ce qui permet de travailler directement depuis le navigateur sans écrire un seul endpoint. Le Row Level Security (RLS) remplace la couche d'authentification serveur. Firebase aurait été viable mais moins familier du SQL ; une API custom aurait cassé la contrainte "zéro serveur".
Pourquoi GitHub Actions pour la galerie plutôt qu'un CMS ?
Le contenu de la galerie est généré, pas édité manuellement. Un CMS aurait introduit un admin à maintenir et une interface dont personne n'avait besoin. Le pipeline Actions est plus élégant : la source de vérité c'est le repo Git. Chaque image générée est tracée, versionnée et le rollback est natif (git revert).
Gestion de la radio : YouTube vs SoundCloud vs Icecast ?
YouTube IFrame API offre un accès gratuit à des milliers d'heures de contenu retrowave déjà curé par la communauté, sans droits à gérer. SoundCloud aurait nécessité des accords artistes. Icecast/Azuracast implique un serveur de streaming dédié — incompatible avec la contrainte static. La contrepartie : dépendance aux politiques YouTube et publicités potentielles.
